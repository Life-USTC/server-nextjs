# 系统设计

这里不再展开文件级说明，而只保留系统层面的设计意图。

## 大体框架

这个项目可以分成四层：

### 产品层

负责定义用户真正关心的对象和任务：班级、时间安排、作业、考试、待办、评论、订阅。

### 页面层

负责把这些对象组织成几个稳定页面角色：

- 首页：判断面
- 详情页：理解对象并采取动作
- 列表页：浏览与筛选
- 管理页：少量配置和维护

### 服务层

负责把页面所需的数据、权限和协议能力拼起来。这里同时承接：

- 站内页面请求
- REST / OpenAPI
- OAuth / OIDC
- MCP

### 数据层

负责保存稳定对象关系，而不是直接决定页面长什么样。Schema 是约束，不是产品本身。

## 为什么这样分

- 产品目标变化慢，页面形式可以调整
- 页面职责变化慢，具体组件可以替换
- 协议边界必须稳定，因为会影响客户端与外部工具
- 数据模型要服务产品，不应反过来绑架页面结构

## 当前技术选择背后的原因

- Next.js：同时承接页面、服务端数据拼装和 API
- Prisma：把对象关系和数据约束集中到同一层
- OAuth / OIDC：把“用户登录”和“第三方客户端访问”分开
- MCP：把工具型调用和普通页面/REST 调用分开

## 代码结构映射

系统分层在仓库里的落点大致如下：

- `src/app`
  - 页面路由
  - Route Handlers
  - `/api-docs` 等站内入口
- `src/features`
  - 面向用户任务的功能域实现
  - 例如首页聚合、作业、待办、评论、上传
- `src/components`
  - 可复用界面组件
  - 包括通用 UI 和少量页面级复合组件
- `src/lib`
  - 基础设施层
  - 例如数据库、鉴权、MCP、日志、存储、时间处理
- `prisma`
  - Schema 与 migrations
- `tools`
  - seed、导入、OpenAPI 后处理等开发脚本
- `tests/e2e`
  - 真实应用级 E2E，用来验证路由、鉴权、页面和 API 的联动

这层映射的目的不是替代代码阅读，而是先帮助你判断“改动应该落在哪一层”，避免一上来就在错误目录里堆逻辑。

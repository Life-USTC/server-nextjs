# 开发规范

这里只保留少量稳定约束，不展开操作手册。

## 基本原则

- 先对齐产品目标和页面职责，再开始写实现
- 先判断是页面问题、协议问题还是数据问题，再决定改哪一层
- 新能力应优先复用现有设计语言和系统边界，不要轻易另起一套

## 实现时需要持续检查的点

- 是否破坏了首页作为“判断面”的定位
- 是否破坏了 `section` 作为核心工作单元的定位
- 是否破坏了评论必须依附对象的原则
- 是否引入了新的鉴权边界或协议边界
- 是否遗漏了国际化处理

## 交付前的最低要求

- 使用 `bun` / `bunx`
- 运行格式和检查：`bun run check --write`
- 运行构建：`bun run build`
- 运行 E2E：`bun run test:e2e`

## 数据与测试

- 新模型或特殊逻辑要补 seed 数据
- 新用户路径要补或更新 E2E
- Prisma schema 是数据约束的最终来源

## 提交规范

- 使用 conventional commit
- 只提交与当前变更直接相关的内容
- 在提交前完成检查、构建和测试

更细的本地、数据库、部署约束以 [`../AGENTS.md`](../AGENTS.md) 为准。
